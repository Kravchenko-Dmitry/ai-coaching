# ADR-004: Centralized Authentication with OpenID Connect

## Status
Accepted

## Context
Each service implementing its own login and token handling led to inconsistent security
and duplicated user stores.

## Decision
All user-facing applications authenticate against the central identity provider (Keycloak)
using OpenID Connect / OAuth 2.0. Services validate JWT access tokens; no service maintains
its own credential store.

## Rules
- Access tokens are short-lived (15 minutes); refresh tokens are handled only by the BFF,
  never exposed to browser JavaScript.
- Service-to-service calls use the client credentials flow.
- Roles and permissions are claims in the token; services must not query the IdP per request.

## Consequences
- Keycloak becomes tier-1 infrastructure with an availability requirement of 99.9%.
- Local development uses a shared dev realm with seeded test users.
